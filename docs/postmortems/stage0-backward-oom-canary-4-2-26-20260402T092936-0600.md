## Most Important Current Inference

The current evidence supports this interpretation:

1. The training graph for step 0 is very large.
2. The router is currently not reducing effective depth on the failing step.
3. The immediate crash still occurs at a very ordinary BF16 matmul with a non-contiguous transposed RHS.
4. The backend/layout path is therefore a more direct explanation than raw capacity.

The best current plain-language diagnosis is:

- the recurrent training graph is stressing a backward-time tensor layout path
- Candle/Burn is attempting to materialize a transposed RHS contiguously during autodiff matmul
- that path is the immediate crash site

This means:

- there is a **near-term primitive/backend problem**
- and there is also a **broader graph-management problem**

Those should not be conflated.

## Signals That Guide the Fix Space

### Signals that favor a backend/layout fix first

- failure occurs at `backward_start`, not during forward
- forward and loss complete successfully
- the failing matmul shape is ordinary: `[2, 1024] x [1024, 1024]`
- the RHS is explicitly non-contiguous before the crash
- GPU memory is still low before the crash

If these continue to hold, the first fix should be a **layout/backward seam fix**, not a change in model ambition.

### Signals that favor a training-loop strategy change second

- `rule_invocations = 17872` in step 0
- `positions_early_exited = 0`
- full-sequence, full-depth unroll is happening in practice

If the primitive/backend issue is repaired and the graph still remains too heavy, the next fix class is likely:

- gradient checkpointing
- truncated BPTT
- stronger router/depth control

These are training-loop changes, not proof that the kernel itself is invalid.

### Signals that would justify reconsidering backprop more fundamentally

Only after the above two layers are addressed should we ask whether ordinary BPTT itself is the wrong training abstraction for the top family.

The most on-thesis alternative, if needed, is not a random new gradient trick but something closer to:

- fixed-point / equilibrium training
- implicit differentiation for a contractive recurrent kernel

That would only be justified if we can show:

- the p1 family behaves like a practical contractive system
- deeper recurrence is mostly convergence behavior
- equilibrium solves are numerically trustworthy

## Relationship to MoR

MoR matters as a control, not as a proof that our current loop shape is correct.

Useful lessons:

- recursive/shared-depth LM pretraining can work with standard backprop
- routing is a primary scientific variable
- efficiency claims must be supported by real training-time behavior, not architectural intuition

What MoR does **not** prove for us:

- that our token-by-token recurrent unroll is already a good BPTT shape
- that our router is currently healthy
- that our backend/layout path is sound

Current run evidence says:

- adaptive depth is not operationally helping on step 0 yet
- we should not claim any compute-efficiency story from the router until it actually exits meaningfully

## Gradient Highway Clarification for the Top 3

For the current top three kernels, the gradient-highway claim is substantively true:

- `p1_fractal_hybrid_composite_v1`
- `p1_fractal_hybrid_v1`
- `p1_contractive_v1`

All three share the p1-family gated residual rule shape:

```text
g * update + (1 - g) * previous_state
```

This makes it fair to say:

- the top three do have an explicit identity-style gradient highway inside the reusable recursive rule
- this is a plausible contributor to why they survived where more explosive families did not

What should **not** be claimed yet:

- that this has “solved” vanishing/exploding gradients at scale
- that no auxiliary stabilization exists
- that the current H200 failure is explained by missing highway behavior

Important repo truth:

- Stage 0 still explicitly uses gradient clipping:
  - `gradient_clip_norm = 1.0`

So the honest position is:

- the top three likely benefit materially from an embedded highway
- but the overall training stack still includes additional stabilization mechanisms

## Assessment of Candidate Fix Classes

### 1. Backend / layout normalization

Current evidence level: **high priority**

Why:

- directly matches the observed failure
- least semantic churn
- should be fixed before we reinterpret the training regime

### 2. Gradient checkpointing

Current evidence level: **plausible second step**

Why:

- addresses activation-memory pressure
- does not directly explain the current transposed-RHS contiguous-copy crash

### 3. Truncated BPTT

Current evidence level: **plausible but semantic**

Why:

- likely helpful if the graph remains too large
- changes long-range credit assignment intentionally
- should be adopted as a deliberate contract choice, not a hidden workaround

### 4. Reversible / adjoint methods

Current evidence level: **not yet**

Why:

- interesting future direction
- not justified as the next practical repair

### 5. Different backprop entirely

Current evidence level: **not yet, but not ruled out**

Most promising on-thesis candidate if ordinary BPTT remains wrong:

- DEQ / implicit fixed-point differentiation for a contractive recurrent family

Least justified current move:

- jumping directly to exotic gradient estimators before fixing the obvious backward/layout primitive

## Immediate Conclusions

1. The canary still failed.
2. The new diagnostics/failure-snapshot work succeeded and was worth doing.
3. We now have a much tighter statement of the bug:
   - backward/layout failure under a very large recurrent graph
4. We should not change model semantics blindly from here.
5. The next repair should target the primitive/backend seam first.

## Recommended Next Moves

1. Extract the backward/layout failure into the narrowest possible reproducible primitive.
2. Identify whether the first actionable repair belongs in:
   - the Candle matmul/backward seam
   - Burn linear usage/layout discipline
   - our recurrent readout / state-mix path
3. Keep the current manifest/hardware/model fixed while verifying the first repair.
4. Only after the primitive failure is repaired:
   - reassess router effectiveness
   - reassess whether checkpointing or TBPTT is necessary
   - reassess whether ordinary BPTT is the correct long-run training abstraction

## Bottom Line

This canary did not greenlight Stage 0.

But it did exactly what a good canary should do:

- fail cheaply
- preserve evidence
- narrow the problem
- keep us from making the next decision blind

The current problem is no longer “something is wrong somewhere in the Stage 0 path.”

It is much closer to:

- “the current backward/layout implementation is not robust under this recurrent graph shape”

That is a hard result, but it is a useful one.
