DREEGMOR Mini-MoE Walkthrough

This page turns the architectures we discussed into animated token-flow diagrams. The sample token string is illustrative only. The goal is to make the structural differences visible before we commit to the next model family.

Reading order: dense baseline, explicit block seam, one-shot MoE block, recurrent MoE block.

1. Dense A Baseline

Every token follows the same shared path through the entire stack. There is no routing decision, no expert selection, and no conditional computation.

2. Explicit Block Seam

This is the reusable block abstraction we need in Rust: shared attention path, then a pluggable FFN slot. The FFN slot is where dense FFN, one-shot MoE FFN, and recurrent MoE FFN can differ without rewriting the rest of the block.

3. Reference Mini-MoE: One-Shot Router

Tokens reach the FFN seam, the router scores experts once, and each token takes a sparse top-1 expert path before the block recombines.

4. Thesis Mini-MoE: Recurrent Router

The recurrent router refines the routing decision over multiple rounds before the final top-1 expert path is chosen. The experts are still feedforward. The recurrence lives in the controller, not in the expert FFNs themselves.

In this illustration, refines and expert change assignments between routing rounds. That is the architectural promise of recurrent routing: more than one chance to decide before committing to an expert.
